# Условия выполнения задачи

Задачи могут завершаться автоматически при выполнении заданного условия, через кнопки в квестовой книге (поле `buttons`) или явной командой.

> Автоматическая проверка условий доступна только для датапак-квестов. Задачи, созданные командами, выполняются исключительно командой `quest complete`.

Условия проверяются каждый тик у задач активного этапа всех активных квестов — закреплённых и незакреплённых.

Для каждой задачи можно задать условие успеха и условие провала:

```json
{
	"tasks": {
		"my_task": {
			"condition": {
				"success": {
					// Описание условия успеха
				},
				"failure": {
					// Описание условия провала
				}
			}
		}
	}
}
```

Оба поля внутри `condition` необязательны — можно задать только одно или оба.

---

## Формат условий

```json
{
	"type": "тип условия"
	// Поля, уникальные для каждого типа
}
```

---

## Условие достижения счёта (`score`)

```json
{
	"type": "score",
	"objective": "<string>",
	"criterion": "<ScoreboardCriterion>",
	"player": "<string>",
	"initial": "<int>",
	"target": "<int>"
}
```

Использует систему scoreboard Minecraft. Под задачей с этим условием в HUD отображается прогресс-бар.

- `objective` — *обязательное*. Название scoreboard objective. Если такого objective нет — он будет создан с типом из `criterion` в момент загрузки задачи.
- `criterion` — тип objective при создании. Используются типы, поддерживаемые Minecraft (полный список на [вики](https://minecraft.wiki/w/Scoreboard#Criteria)). По умолчанию: `"dummy"`.
- `player` — никнейм игрока, чей счёт проверяется. Поддерживает виртуальных игроков (как в scoreboard), но не поддерживает селекторы. Если не указан — проверяется счёт игрока, выполняющего квест.
- `initial` — счёт, который будет установлен игроку в момент загрузки задачи. Если не указан — счёт не перезаписывается (используется текущее значение игрока в objective).
- `target` — *обязательное*. Значение, которого должен достичь счёт. Если указан `initial` и он **больше** `target` — условие работает в обратном направлении: нужно опуститься ниже `target`.

### Примеры

#### Вскопать 5 дубовых брёвен
```json
{
	"type": "score",
	"objective": "oak_logs_mined",
	"criterion": "minecraft.mined:minecraft.oak_log",
	"target": 5
}
```

#### Уничтожить 10 000 зомби всем сервером
Потребуются дополнительные приготовления через хуки жизненного цикла (см. [ХУКИ ЖИЗНЕННОГО ЦИКЛА ЗАДАЧИ](/datapack-reference/task-lifecycle-hooks.md)).

```json
{
	"type": "score",
	"objective": "zombies_killed_total",
	"player": "GLOBAL",
	"target": 10000
}
```

## Условие выполнения предиката (`predicate`)

```json
{
	"type": "predicate",
	"predicate": "<Identifier>"
}
```

Использует систему predicate Minecraft. Предикаты — это условия, описанные в JSON-файлах датапака. Задача выполнится, когда выполнится предикат. Бинарное условие: нет прогресс-бара.

---

## Составное условие (`all`)

Позволяет объединить несколько условий — задача выполнится только когда все они выполнены. Прогресс в HUD = число выполненных подусловий.

```json
{
	"type": "all",
	"conditions": [
		{ "type": "..." },
		{ "type": "..." }
	]
}
```

Условия внутри `all` могут быть любого типа и вложены друг в друга без ограничений.

### Пример

```json
{
	"type": "all",
	"conditions": [
		{
			"type": "score",
			"objective": "gold_collected",
			"target": 100
		},
		{
			"type": "predicate",
			"predicate": "example:has_key"
		}
	]
}
```

---

## Составное условие (`any`)

Задача выполнится, когда выполнено **хотя бы одно** из подусловий. Бинарное: нет прогресс-бара.

```json
{
	"type": "any",
	"conditions": [
		{ "type": "..." },
		{ "type": "..." }
	]
}
```

---

## Составное условие (`none`)

Задача выполнится, когда **ни одно** из подусловий не выполнено. Бинарное: нет прогресс-бара.

```json
{
	"type": "none",
	"conditions": [
		{ "type": "..." },
		{ "type": "..." }
	]
}
```

---

## Условие по статусу задач (`tasks`)

Проверяет, сколько задач **того же квеста** имеют нужный статус.

```json
{
	"type": "tasks",
	"tasks": ["task_id_1", "task_id_2"],
	"status": "success",
	"count": 1
}
```

- `tasks` — список task ID для проверки. Если не указан или пустой — используются все задачи **активного этапа**. Можно указывать задачи из других этапов.
- `status` — ожидаемый статус: `"success"`, `"failure"` или `"skipped"`. Если не указан — засчитывается задача в **любом терминальном** статусе (success, failure или skipped).
- `count` — минимальное число задач с нужным статусом. Если не указан — должны совпасть **все** задачи из пула.

Если цель больше 1 — отображается прогресс-бар в HUD (число выполненных задач / цель).

### Примеры

#### Оба задания должны быть выполнены успешно

Обязательная задача этапа завершится автоматически, когда обе опциональные выполнены:

```json
{
	"type": "tasks",
	"tasks": ["side_task_a", "side_task_b"],
	"status": "success"
}
```

#### Выполнить хотя бы 2 из 3 заданий

```json
{
	"type": "tasks",
	"tasks": ["task_a", "task_b", "task_c"],
	"status": "success",
	"count": 2
}
```

#### Все задачи текущего этапа должны завершиться (в любом статусе)

```json
{
	"type": "tasks"
}
```

## Кнопки ручного завершения (`buttons`)

Позволяют игроку самостоятельно завершить задачу прямо из квестовой книги. Это альтернатива `condition` — вместо автоматической проверки дизайнер даёт игроку явный выбор.

```json
{
	"tasks": {
		"choose": {
			"title": "Make a choice",
			"buttons": ["success", "failure", "skip"]
		}
	}
}
```

Допустимые значения: `"success"`, `"failure"`, `"skip"`. Порядок в массиве не важен — в книге кнопки всегда отображаются в порядке success → failure → skip. Дубликаты схлопываются (с предупреждением в лог).

Кнопки рисуются под задачей в квестовой книге. При наведении отображается tooltip. По нажатию сервер проверяет, что задача активна и не завершена, после чего выполняет её с выбранным статусом.

`buttons` и `condition` независимы — их можно использовать вместе. Кнопки срабатывают независимо от состояния условия.
